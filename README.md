Перейдіть у папку backend і створіть файл .env на основі .env.example.

Заповніть параметри бази даних:

DB_CONNECTION=pgsql
DB_HOST=invoice_db
DB_PORT=5432
DB_DATABASE=invoice_db
DB_USERNAME=user
DB_PASSWORD=password
Запуск проєкту

Побудуйте образи та запустіть контейнери:

docker compose up -d --build

Після першого запуску виконайте міграції:

docker compose exec backend php artisan migrate

1. Як структурував frontend і backend?
Backend: Використано архітектуру Thin Controllers. Бізнес-логіку розділено на сервісні класи: InvoiceQueryService (читання) та InvoiceManagementService (запис/оновлення). Дані передаються через API Resources.

Frontend: Використано компонентний підхід Nuxt 4. Валідація форм реалізована за допомогою vee-validate та zod для забезпечення суворої типізації та зручного UX.

2. Які компроміси зробив і чому?
Розрахунок gross_amount виконується на стороні клієнта для швидкого відгуку інтерфейсу, проте на бекенді обов'язково проводиться вторинна перевірка даних для безпеки.
Життєвий цикл статусу інвойсу: Згідно з ТЗ, інвойси створюються у статусі `pending`, а редагування дозволено виключно для цього статусу. Оскільки в завданні відсутні вимоги щодо ендпоінтів для затвердження/відхилення документів (`approve` / `reject`), у поточному MVP-модулі інвойси залишаються у статусі `pending`.
Відсутність повноцінної системи аутентифікації: Згідно з ТЗ, аутентифікація не була потрібна, тому було сфокусовано зусилля на безпеці бізнес-логіки (валідація станів інвойсів) замість створення механізмів авторизації.

3.Що б ти покращив у production-версії?
Додав би автоматизовані юніт-тести (Pest/PHPUnit) 
Додав би кешування запитів до бази даних для часто отримуваних інвойсів та індексацію полів, за якими здійснюється пошук.

4. Які UX edge cases ти врахував?
Блокування редагування: Форма динамічно переходить у стан "read-only", якщо статус інвойсу відмінний від pending, запобігаючи несанкціонованим змінам затверджених документів.

Миттєвий фідбек: Реалізовано валідацію сум у реальному часі: gross_amount автоматично перераховується і відображається користувачеві, а помилки валідації Zod відображаються під полями одразу після введення.

Запобігання помилкам дат: Враховано логіку перевірки due_date >= issue_date, що запобігає створенню логічно некоректних рахунків, де термін оплати передує даті виписки.

States: Враховано стани loading (завантаження даних, відправка форми) та error (обробка 422/500 помилок API), щоб користувач завжди розумів, що відбувається з його дією.

Empty state: Додано окремий вигляд для списку, коли інвойсів у системі ще немає.
